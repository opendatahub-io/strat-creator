# Human Review Guide

> **Audience**: Staff engineers, SMEs, and architects reviewing strategies after CI processing.
>
> | Label | Meaning |
> |-------|---------|
> | `strat-creator-rubric-pass` | CI approved — ready for improvements and human sign-off |
> | `strat-creator-needs-attention` | CI flagged — needs fixes before sign-off |
> | `strat-creator-human-sign-off` | Human signed off — feature-ready |

## Which RFEs Enter the Pipeline

Not every RHAIRFE issue gets a strategy. The pipeline selects RFEs that pass two gates:

1. **Scope gate** — the RFE is tagged for the current release cycle:
   - Has the `strat-creator-3.5` or `strat-creator-3.6` label, **OR**
   - Has a Target Version of `rhoai-3.5`, `rhoai-3.5.EA1`, `rhoai-3.5.EA2`, `rhoai-3.6`, `rhoai-3.6.EA1`, `rhoai-3.6.EA2`, `3.5 GA RHOAI RELEASE`, `3.6 EA1 RHAII RELEASE`, `3.6 EA1 RHOAI RELEASE`, `3.6 EA2 RHOAI RELEASE`, `3.6 GA RHAII RELEASE`, or `3.6 GA RHOAI RELEASE`
2. **Quality gate** — the RFE has been validated:
   - `rfe-creator-autofix-rubric-pass` (CI-approved), **OR**
   - `tech-reviewed` (human-approved)

RFEs in Closed, Resolved, or Draft status are excluded. The filter configuration lives in [`config/pipeline-settings.yaml`](../config/pipeline-settings.yaml).

## Two Paths to Sign-Off

Every strategy arriving here has been through the CI pipeline but still requires a human in the loop. The goal is the same in both cases: **sign off the strategy so it becomes a feature.** The path depends on CI's verdict:

- **Rubric-pass** — CI approved the strategy. You review it, improve it if needed, and then sign off.
- **Needs-attention** — CI flagged issues. You fix the inputs, resubmit through CI, and sign off once it passes.

Both paths end with `/strategy-signoff`, which marks the strategy as feature-ready.

## Scoring

CI scores every strategy on 4 dimensions (Feasibility, Testability, Scope, Architecture), each 0–2.

| Verdict | Trigger | Meaning |
|---------|---------|---------|
| **APPROVE** | total >= 6, no zeros | Ready for human sign-off |
| **REVISE** | total >= 3, at most 1 zero | Fixable — a dimension or two scored low |
| **REJECT** | total < 3 or 2+ zeros | Fundamental problems |

## Setup

1. Install [Claude Code](https://docs.anthropic.com/en/docs/claude-code)
2. Set Jira credentials:
   ```bash
   export JIRA_SERVER="https://redhat.atlassian.net"
   export JIRA_USER="your-email@redhat.com"
   export JIRA_TOKEN="your-atlassian-api-token"  # https://id.atlassian.com/manage-profile/security/api-tokens
   ```
3. Clone the repo: `git clone https://github.com/opendatahub-io/strat-creator.git && cd strat-creator`

## Workflow

### 1. Pull the strategy

```bash
claude "/strategy-pull RHAISTRAT-NNNN"
```

This fetches into `local/` (gitignored, isolated from CI):
- `local/strat-tasks/RHAISTRAT-NNNN.md` — the strategy document you will review and work on
- `local/strat-originals/RHAIRFE-NNNN.md` — the original RFE for reference
- `local/strat-reviews/RHAISTRAT-NNNN-review.md` — the Strat Creator review

> **Large strategies:** When a strategy exceeds Jira's description size limit (~32K chars), the full strategy is stored as a Jira attachment (`RHAISTRAT-NNNN-strategy.md`) and the Jira description shows a TL;DR stub with a note pointing to the attachment. This is handled transparently — `/strategy-pull` reassembles the full content into your local file, and `/strategy-push` and `/strategy-signoff` automatically decide whether to use an attachment based on size.

### 2. Read the strategy

Open `local/strat-tasks/RHAISTRAT-NNNN.md`. The document has three sections:

- **Top section** — a verbatim copy of the original RFE (the business need).
- **`## Strategy (AI Generated by Agentic SDLC Pipeline)`** — the AI-generated "how." Do not edit this directly — it gets regenerated by `/strategy-refine`.
- **`## Staff Engineer / SME Input`** — where you add your changes, corrections, and guidance.

### 3. Read the review

Open `local/strat-reviews/RHAISTRAT-NNNN-review.md` or browse the [dashboard](https://strat-dashboard-0f1209.gitlab.io/). Check:
- **Score table** — which dimensions failed?
- **Prose reviews** — what did each reviewer flag?

### 4. Fix inputs (needs-attention only)

**Never edit strategy text directly.** Edit the inputs, rerun the pipeline, and let it regenerate the output.

| Fix Path | When | What to Edit |
|----------|------|-------------|
| **Overlay** (recommended) | Wrong dependencies, outdated versions, component gaps affecting multiple strategies | Create an overlay in a local checkout of `opendatahub-io/architecture-context`, test it with `/strategy-refine`, then submit a PR upstream. See the [Overlays README](https://github.com/opendatahub-io/architecture-context/blob/main/overlays/README.md) for format. |
| **Staff Engineer / SME Input** | Issue specific to this strategy (effort estimate, scope, test criteria) | `## Staff Engineer / SME Input` section in `local/strat-tasks/RHAISTRAT-NNNN.md` |
| **Architecture context** | Major structural changes (new components, boundary changes) | Contact maintainers (James Tanner, Kevin Bader, Luca Burgazzoli) |

**Prefer overlays.** They fix the root cause for all future strategies, not just the one you're reviewing. Use Staff Engineer / SME Input only for one-off fixes.

**Testing overlays locally before pushing.** You can test overlays with a local checkout of `opendatahub-io/architecture-context` by passing `--architecture-context` to refine and review (see step 5). **Don't forget to submit a PR to `opendatahub-io/architecture-context` afterward** — the overlay only benefits other strategies and reviewers once it's merged upstream.

Example Staff Engineer / SME Input:

```markdown
## Staff Engineer / SME Input

Effort estimate is too low. This touches the inference controller
and the model registry, two teams need to coordinate. Estimate
should be M not S. Add a dependency on RHAISTRAT-1200.
```

### 5. Rerun locally

**Always refine before review.** Refine regenerates the Strategy section (the HOW) incorporating your Staff Engineer / SME Input and any new overlays. Review scores what refine produces — running review without refining first scores the old output.

**Always specify the strategy key.** This ensures you refine and review only the strategy you're working on, not everything in `local/`:

```bash
claude "/strategy-refine RHAISTRAT-NNNN"
claude "/strategy-review RHAISTRAT-NNNN"
```

If you're testing a local overlay, pass `--architecture-context` to both commands:

```bash
claude "/strategy-refine RHAISTRAT-NNNN --architecture-context /path/to/local/architecture-context"
claude "/strategy-review RHAISTRAT-NNNN --architecture-context /path/to/local/architecture-context"
```

Check the updated strategy in `local/strat-tasks/` and scores in `local/strat-reviews/`. Repeat from step 2 if needed.

#### Visual diff

Each time `/strategy-refine` runs, a version history is saved under `local/strat-history/RHAISTRAT-NNNN/`:

```
local/strat-history/RHAISTRAT-NNNN/
├── v0.md              # baseline from /strategy-pull
├── v0-to-v1.html      # word-level HTML diff
├── v1.md              # after first refine
├── v1-to-v2.html
├── v2.md              # after second refine
└── ...
```

Open any `vN-to-vM.html` file in a browser to see a word-level diff of the Strategy section between versions. Additions are highlighted in green, deletions in red. The diff focuses on the Strategy section only (the part that `/strategy-refine` regenerates), so Staff Engineer / SME Input changes are not shown.

The `latest_diff` field in the strategy frontmatter points to the most recent diff file, so you can find it without browsing the history directory.

When you `/strategy-pull` again (e.g., after CI re-evaluates), the existing history is archived to a timestamped folder (`local/strat-history/RHAISTRAT-NNNN-YYYYMMDD-HHMMSS/`) and a fresh v0 baseline is saved. Previous diffs are preserved in the archive.

### 6. Push or sign off

| CI Verdict | Command | What Happens |
|------------|---------|-------------|
| `needs-attention` | `/strategy-push RHAISTRAT-NNNN` | Pushes fixes to Jira, removes `needs-attention`. CI re-evaluates. Come back to sign off after CI approves. |
| `rubric-pass` (want CI re-evaluation) | `/strategy-push RHAISTRAT-NNNN` | Pushes edits to Jira, removes `rubric-pass`. CI re-evaluates with your changes. Come back to sign off after CI approves. |
| `rubric-pass` (ready to finalize) | `/strategy-signoff RHAISTRAT-NNNN` | Pushes strategy content to Jira, posts review summary comment, attaches full review file, adds `strat-creator-human-sign-off`. Strategy is feature-ready. |

#### Pushing CI-approved strategies for re-evaluation

When a strategy already has `rubric-pass` but you've made significant changes (e.g., adding Staff Engineer / SME Input with architectural direction from a KEP), you can push those edits back to Jira and have CI re-evaluate them before signing off. This is useful for:

- **Collaborating with teammates** — push your SME edits so teammates can see the updated strategy in Jira and the dashboard before you sign off.
- **Validating significant changes** — when your input changes the strategy substantially, CI re-evaluation confirms the updated version still passes the rubric.
- **Iterating with CI feedback** — push, let CI re-refine with your input, pull the result, review, and sign off when satisfied.

The workflow: `/strategy-push` removes the `rubric-pass` label, CI picks up the strategy on its next run, re-refines incorporating your Staff Engineer / SME Input, and re-scores. Once it passes again, pull and `/strategy-signoff`.

## Key Rules

1. **Edit inputs, not outputs.** The strategy text is regenerated by refine — direct edits get overwritten.
2. **Prefer overlays over one-off fixes.** They're durable and benefit all future strategies.
3. **Always rerun refine before review.** Review scores what refine produces, not what you wrote.
4. **Push resubmits to CI.** `/strategy-push` works for both `needs-attention` and `rubric-pass` strategies — it removes the current CI label so CI re-evaluates. Use `/strategy-signoff` after CI approves.
5. **Sign-off requires CI approval.** `strat-creator-human-sign-off` requires `strat-creator-rubric-pass`.
